    Python with Virtual Environments (venv)¶

    What you'll learn

    How to create, activate, and use Python virtual environments on Discovery for reproducible, isolated Python workflows.

    Why Use a venv on the Cluster?¶

    The system Python on Discovery is shared and locked down — you can't pip install into it. Virtual environments give you an isolated space to install exactly the packages you need without conflicting with other users or system packages.

    Don't use pip install --user

    While --user installs technically work, they pollute your home directory and create version conflicts across projects. Always use a venv instead.

    Step-by-Step¶

    1. Load a Python Module¶

    First, check what Python versions are available:

    module avail python

    Load the version you want:

    module load python/3.11  # (1)!
    1. Check module avail python for the latest available versions.

    2. Create the Virtual Environment¶

    python -m venv ~/envs/myproject  # (1)!
    1. Store venvs in ~/envs/ to keep things organized. You can also place them in your project directory.

    3. Activate and Install Packages¶

    source ~/envs/myproject/bin/activate
pip install --upgrade pip
pip install numpy pandas scikit-learn

    Pin your dependencies

    Always save your requirements for reproducibility: 

    pip freeze > requirements.txt

    4. Use in a Batch Job¶

    #!/bin/bash
#SBATCH --job-name=python_venv
#SBATCH --partition=standard
#SBATCH --time=02:00:00
#SBATCH --cpus-per-task=4
#SBATCH --mem=8G
#SBATCH --output=%x_%j.out
#SBATCH --error=%x_%j.err

# Load required modules
module load python/3.11

# Run your work
source ~/envs/myproject/bin/activate
python my_analysis.py

    Submit it:

    sbatch run_analysis.sh

    Common Pitfalls¶

    ModuleNotFoundError in batch job

    You probably forgot to activate the venv in your sbatch script. Make sure source ~/envs/myproject/bin/activate appears after module load and before your python command.

    Disk quota exceeded when installing packages

    Your home directory has a quota. For large environments (especially ML packages), create the venv on scratch storage instead: 

    python -m venv /scratch/$USER/envs/myproject
    Remember that scratch is not backed up — keep your requirements.txt in version control.

    Wrong Python version inside the venv

    The venv is tied to the Python that created it. If you loaded python/3.9 when you created it, the venv uses 3.9 regardless of what module is currently loaded. Create a new venv if you need a different version.
